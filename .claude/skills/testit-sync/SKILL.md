# Выгрузка тест-кейсов в Test IT

Соответствует этапу «Написание тест-кейсов» в Shift Left (`docs/Testirovanie-Shift-Left.md`).

## Подготовка

**Важно про `.env`:** `Read(.env)` запрещён правилом `permissions.deny` в
`.claude/settings.json` — токен никогда не должен попадать в контекст агента как
текст. Вместо чтения файла инструментом Read всегда загружай `.env` в переменные
окружения PowerShell-скриптом и работай с ними только внутри PowerShell.

**Важно про сами переменные окружения:** состояние PowerShell (в т.ч. переменные
окружения) **не сохраняется между отдельными вызовами** инструмента — сохраняется
только рабочая директория. Поэтому загрузку `.env` нельзя сделать один раз в начале:
её нужно повторять в **каждой** PowerShell-команде этого навыка, которая обращается к
Test IT (проверка подключения, поиск секции, создание каждого кейса). Используй этот
блок как первую часть любой такой команды:

```powershell
if (-not (Test-Path ".env")) { throw "ENV_MISSING" }
Get-Content ".env" | Where-Object { $_ -match '=' -and $_ -notmatch '^\s*#' } | ForEach-Object {
  $k, $v = $_ -split '=', 2
  Set-Item -Path "env:$($k.Trim())" -Value $v.Trim()
}
```

1. Проверь наличие и заполненность переменных без вывода значений в чат:
   ```powershell
   if (-not (Test-Path ".env")) { throw "ENV_MISSING" }
   Get-Content ".env" | Where-Object { $_ -match '=' -and $_ -notmatch '^\s*#' } | ForEach-Object {
     $k, $v = $_ -split '=', 2
     Set-Item -Path "env:$($k.Trim())" -Value $v.Trim()
   }
   $missing = @('TESTIT_URL','TESTIT_TOKEN','TESTIT_PROJECT_ID') | Where-Object {
     [string]::IsNullOrWhiteSpace((Get-Item "env:$_" -ErrorAction SilentlyContinue).Value)
   }
   if ($missing) { "ENV_INCOMPLETE: $($missing -join ', ')" } else { "ENV_OK" }
   ```
   Если `ENV_MISSING`/`ENV_INCOMPLETE` — остановись и попроси тестировщика заполнить
   `.env` по образцу `.env.example`. **Значения переменных никогда не выводи в чат** —
   команда выше и так не печатает их, не меняй это при адаптации.
2. Определи, какие кейсы выгружать: файл из `results/test-cases/`, указанный тестировщиком,
   или последний созданный в этой сессии. Блок «Самопроверка агента» в конце файла — это
   внутренняя отметка навыка `test-case-design`, не кейс; в Test IT её не создавай.

## Проверка подключения (обязательна перед созданием)
Авторизация во всех запросах — заголовок `Authorization: PrivateToken <TESTIT_TOKEN>`.
Загрузку `.env` (блок выше) включай в начало этой же команды:

```powershell
Get-Content ".env" | Where-Object { $_ -match '=' -and $_ -notmatch '^\s*#' } | ForEach-Object {
  $k, $v = $_ -split '=', 2
  Set-Item -Path "env:$($k.Trim())" -Value $v.Trim()
}
Invoke-RestMethod -Uri "$env:TESTIT_URL/api/v2/projects/$env:TESTIT_PROJECT_ID/sections" `
  -Headers @{ Authorization = "PrivateToken $env:TESTIT_TOKEN" }
```

Если запрос падает — покажи текст ошибки (без токена) и остановись. Не подбирай
эндпоинты и названия статусов/полей наугад: если API инстанса отличается от того, что
описано ниже, попроси тестировщика прислать ссылку на Swagger своего Test IT
(`<TESTIT_URL>/api/swagger`) и сверься с ним, прежде чем создавать кейсы.

## Структура: секция и сьют
По регламенту компании (`docs/Testirovanie-Shift-Left.md`):
- **Название секции** должно совпадать с названием User Story;
- **Сьюты внутри секции** соответствуют названиям задач, на которые декомпозирована US.

Перед созданием кейсов найди (по названию, через `GET /sections`) или, если её нет и
тестировщик подтвердил создание, создай нужную секцию/сьют. Если из контекста неясно,
к какой US/задаче относятся кейсы, — спроси тестировщика, не придумывай название.

## Создание кейса
Эндпоинт: `POST <TESTIT_URL>/api/v2/workItems`. Маппинг полей шаблона
(`templates/test-case-template.md`, правила — `docs/Napisanie-test-kejsov.md`):

| Поле шаблона | Поле API | Комментарий |
|---|---|---|
| Заголовок | `name` | |
| Предусловия | `preconditionSteps` (массив `{ action }`) | |
| Шаги + ожидаемый результат | `steps` (массив `{ action, expected }`) | |
| Приоритет | `priority`: `Lowest`/`Low`/`Medium`/`High`/`Highest` | соответствует Самый низкий…Самый высокий |
| Статус | `state` | Не готов / Готов / Требует доработки — **точные значения enum сверь со Swagger**, ниже — предположение |
| Статус автоматизации | поле автоматизации в API (уточни по Swagger) | по умолчанию «Не автоматизирован» |
| Тег (раздел/страница) | `tags` (массив `{ name }`) | mainPage, searchPage и т.п. |
| Требование (ФТ-*) | `tags` | отдельный тег для трассируемости (схема БТ/ФТ/ТК — см. `checklists/requirements-checklist.md`, п.0.3) |
| Сценарий (позитивный/негативный) | `tags` | название поля в шаблоне — «Сценарий», не «Тип» |

Обязательные поля тела: `entityTypeName: "TestCases"`, `projectId`, `sectionId` (секция US,
см. выше), `state` (по умолчанию соответствует статусу «Не готов»), `name`, `steps`.
Если название поля статуса/статуса автоматизации в Swagger отличается от предположения
выше — используй то, что реально есть в API, а не то, что здесь написано.

## Порядок
1. Покажи тестировщику план: сколько кейсов, в какие проект/секцию/сьют. Дождись подтверждения.
2. Создавай кейсы по одному; **каждый вызов `Invoke-RestMethod` — отдельная PowerShell-команда
   с загрузкой `.env` в её же начале** (см. «Подготовка» — переменные окружения не переживают
   переход к следующему вызову инструмента). После каждого созданного кейса фиксируй `id` и
   `globalId` из ответа.
3. Итог — таблица в чате: `TC-* → globalId → ссылка <TESTIT_URL>/projects/<PROJECT_ID>/tests?workItemId=<id>`.
4. Ошибки не замалчивай: если часть кейсов не создалась, перечисли какие и почему.

## Правила
- Не изменяй и не удаляй существующие сущности в Test IT — только создание новых.
- Один прогон — одна пачка кейсов; не повторяй запросы при неоднозначных ошибках (риск дублей).
- Приоритет «Самый высокий» у кейсов по рисковым сценариям не понижай при выгрузке.
- Никогда не используй Read/Grep/Glob на `.env` — только PowerShell-загрузку в переменные
  окружения, описанную в «Подготовка» (это же теперь закрыто и `permissions.deny` в
  `.claude/settings.json`).
