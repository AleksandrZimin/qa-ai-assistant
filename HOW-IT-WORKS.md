# Как устроен агент

Документ для тестировщиков и тех, кто будет развивать пространство.

## Уровни инструкций

| Уровень | Файлы | Для кого |
|---|---|---|
| 1. Правила поведения | `CLAUDE.md` | Агент. Загружается в каждую сессию: роль, запрет фантазировать, порядок поиска ответов, экономия токенов, карта проекта |
| 2. Навыки (процедуры) | `.claude/skills/*/SKILL.md` | Агент. Пошаговые алгоритмы конкретных задач; подключаются автоматически, когда промпт похож на задачу навыка |
| 3. Руководство | `README.md`, `HOW-IT-WORKS.md` | Тестировщики |

## Навыки агента

Основной контур — QA-процесс компании:

| Навык | Что делает | Опирается на |
|---|---|---|
| `testing-requirements` | Прогоняет требования через каждый пункт чек-листа (структуру и формат итога задаёт сам чек-лист — сейчас разделы 0+А–Ж, статусы ✅⚠️❌➖, блокирующие 🔴), перед сдачей — обязательная самопроверка → `results/requirements-review/` | `checklists/requirements-checklist.md`, `templates/Otchet-o-testirovanii-trebovanij.md` |
| `test-case-design` | Выделяет реестр требований по схеме БТ/ФТ (та же, что в чек-листе), на каждое ФТ — позитивные и негативные кейсы (ТК-<ФТ>.<n>) по правилам компании, перед сдачей — обязательная самопроверка → `results/test-cases/` | `templates/test-case-template.md`, `templates/test-case-example.md` (формат таблицы трассировки), `docs/Napisanie-test-kejsov.md` |
| `testit-sync` | Создаёт кейсы в Test IT через REST API: секция = US, сьют = задача (сначала проверка подключения, потом подтверждение плана у тестировщика) | `.env`, `docs/Testirovanie-Shift-Left.md` |
| `save-context` | Сохраняет выжимку сессии в `.context-cache/` | — |

### Самопроверка перед сдачей

Идея подсмотрена в [claude-code-orchestrator-kit](https://github.com/maslennikov-ig/claude-code-orchestrator-kit)
(там — `validate-report-file`/`run-quality-gate` для кода) и адаптирована под ручное QA:
`testing-requirements` и `test-case-design` не показывают результат тестировщику сразу —
сначала механически сверяют черновик со спецификацией (все ли пункты чек-листа закрыты,
у каждого ли требования есть позитив+негатив, нет ли зашитых статических данных и т.д.) и
исправляют найденное. В конце сохранённого файла остаётся короткий блок «Самопроверка
агента» — не для копирования в YouTrack/Test IT, только для прозрачности тестировщику.

Справочные скиллы — большие адаптированные материалы (изначально написаны для другой
платформы, Perplexity Computer; конвертированы в формат Claude Code). Агент применяет их
только когда тестировщик явно выходит за рамки ручного тестирования, не проактивно:

| Навык | Когда применяется |
|---|---|
| `dev-engineering` | Код, архитектура, автоматизация, CI/CD, security |
| `product-management` | PRD, acceptance criteria, приоритизация, роадмап |
| `research-knowledge` | Глубокое исследование, анализ данных, оценка источников |
| `token-efficient` | Фоновый стиль ответов — краткость, без воды (дублирует часть правил CLAUDE.md как подробный справочник) |

Каждый такой файл начинается с пояснения об адаптации: команды вида `python scripts/*.py`
внутри — иллюстративные паттерны из первоисточника, в репозитории их нет и вызывать их
не нужно.

## Модель local/ vs docs/ (матричная структура)

Компания работает в матричной QA-структуре (`docs/Operacionnaya-model-QA.md`, разделы
5, 7, 8): центральный QA задаёт единые стандарты, продуктовые команды адаптируют их под
свой домен. Это отражено в двух слоях контекста:

| Слой | Папки | В git? | Кто наполняет |
|---|---|---|---|
| Общие стандарты компании | `docs/`, `checklists/`, `templates/` | Да | Центральный QA |
| Локальная адаптация команды | `local/docs/`, `local/checklists/`, `local/templates/` | Нет (кроме README.md) | Продуктовая команда/проект |

Правило переопределения: файл в `local/...` с **тем же именем**, что в общей папке,
полностью замещает общий для этой команды при чтении агентом. Файл с новым именем —
просто добавляется к контексту (например, документация конкретного продукта). Подробности
и обоснование — в `local/README.md`.

Так разные продуктовые команды, проекты и даже разные компании могут клонировать один и
тот же шаблон, получать общие обновления методологии и при этом не конфликтовать
друг с другом локальными адаптациями.

## Порядок поиска ответа (локальный контекст)

На любой вопрос агент ищет ответ в таком порядке:

1. `local/docs/` — документация и адаптации конкретной продуктовой команды (приоритет);
2. `docs/` — общие стандарты компании (если в `local/docs/` нет нужного файла/ответа), см. `docs/README.md`;
3. `.context-cache/` — итоги сессий за последние 2 дня;
4. Общие знания (тестирование, разработка, менеджмент) — с пометкой 🧠;
5. Вопрос тестировщику — если ответа нет нигде. Выдумывать агенту запрещено.

Аналогично для чек-листа и шаблонов: `local/checklists/` / `local/templates/` проверяются
раньше `checklists/` / `templates/` — это уже встроено в навыки `testing-requirements` и
`test-case-design`.

## Ограничение: нет интеграции с YouTrack/Teams

Реальный workflow компании (`docs/Testirovanie-Shift-Left.md`, `docs/Testirovanie-Shift-Right.md`)
предполагает публикацию отчётов комментарием в задаче YouTrack и коммуникацию в Teams.
У агента есть интеграция только с Test IT (через `.env`). Отчёты по требованиям и другие
артефакты агент готовит в файлах `results/` в формате, который можно скопировать в
YouTrack как есть — саму публикацию выполняет тестировщик вручную.

## Кэш контекста

- После крупной задачи (или по промпту «сохрани контекст») агент пишет короткую
  выжимку в `.context-cache/`: что делали, решения, незакрытые вопросы.
- При старте каждой сессии хук `SessionStart` (`.claude/settings.json`) запускает
  `.claude/scripts/clean-cache.ps1`, который удаляет файлы старше 2 дней.
- Кэш локальный (в `.gitignore`) — у каждого тестировщика свой.
- Скрипт очистки написан под Windows PowerShell; на macOS/Linux нужен `pwsh`
  или замена команды хука на shell-аналог.

## Язык общения (RU по умолчанию, переключение на EN)

`.claude/settings.json` задаёт `language: "russian"` как предпочтение харнесса, а
`CLAUDE.md` — явное правило переключения: если тестировщик пишет на английском или
прямо просит («switch to English»), агент ведёт диалог и пишет файлы результатов
(`results/`, `.context-cache/`) на английском, пока не попросят вернуться на русский.

Два уровня инструкций (README/HOW-IT-WORKS) переведены как отдельные файлы
(`README.en.md`, `HOW-IT-WORKS.en.md`) — это статичная документация для человека,
дублировать её проще, чем поддерживать перевод «на лету». Методология же
(`docs/`, `checklists/`, `templates/`) остаётся **единственным источником на языке
оригинала** — заводить `docs-en/`/`checklists-en/` не стали: агент прекрасно понимает
русский вход независимо от того, на каком языке отвечает, а параллельные копии
методологии означали бы двойную поддержку и риск разъехаться при правках. Английская
копия существует только для инструкций верхнего уровня, а не для чек-листа/шаблонов —
их редактируем только на русском.

## .claude/settings.json

Помимо хука очистки кэша, файл фиксирует:
- `language: "russian"` — язык ответов на уровне харнесса по умолчанию (мягкое
  предпочтение — `CLAUDE.md` явно разрешает переключение на английский по просьбе,
  см. раздел выше);
- `autoUpdatesChannel: "stable"` — тестировщики не разработчики, им не нужен bleeding edge;
- `permissions.deny: ["Read(.env)"]` — агенту запрещено читать `.env` инструментом Read.
  Токен Test IT должен попадать только в переменные окружения PowerShell, а не в текст,
  который видит модель (см. ниже и `testit-sync/SKILL.md`).

**Важный нюанс для любого скилла, работающего с внешним API через `.env`:** состояние
PowerShell (переменные окружения, функции) **не сохраняется между отдельными вызовами**
инструмента — сохраняется только рабочая директория. Значит, `.env` нужно загружать в
переменные окружения заново в начале **каждой** PowerShell-команды, которая обращается к
API, а не один раз в начале скилла. `testit-sync` уже учитывает это; если добавляете новую
интеграцию — повторите этот же паттерн, а не полагайтесь на «загрузил один раз в начале».

## Экономия токенов и точность

Зашиты в `CLAUDE.md` как обязательные правила:
- поиск по файлам (Grep) вместо чтения целиком; большие документы — по частям;
- объёмные результаты — в файлы `results/`, в чат — только сводка;
- каждое утверждение о требованиях — со ссылкой `файл:строка`;
- неоднозначность — повод для вопроса, а не молчаливой интерпретации.

## Как добавить новое умение (нагрузка, автоматизация, …)

1. Создайте папку `.claude/skills/<имя-навыка>/` с файлом `SKILL.md`.
2. Во frontmatter: `name` и `description` — по description агент решает, когда
   применять навык, поэтому перечислите в нём типичные формулировки промптов.
3. В теле: входные данные, пошаговый алгоритм, формат результата, правила.
   Смотрите существующие навыки как образец.
4. Опорные файлы (шаблоны, чек-листы) кладите в `templates/` или `checklists/`
   и ссылайтесь на них из навыка по относительному пути. Если файл может отличаться
   у разных продуктовых команд — научите навык сначала проверять одноимённый файл в
   `local/templates/` / `local/checklists/` (см. модель `local/` vs `docs/` выше).
5. Допишите строку о новом процессе в таблицу «Рабочие процессы» в `CLAUDE.md`
   и в таблицу возможностей в `README.md`.

## Структура проекта

```
project-root/
├── CLAUDE.md                  # инструкции агента (уровень 1)
├── README.md                  # быстрый старт для тестировщиков
├── HOW-IT-WORKS.md            # этот файл
├── .env.example               # образец секретов Test IT (.env — в .gitignore)
├── .claude/
│   ├── settings.json          # хук автоочистки кэша
│   ├── scripts/clean-cache.ps1
│   └── skills/                # навыки агента (уровень 2)
│       ├── testing-requirements/
│       ├── test-case-design/
│       ├── testit-sync/
│       ├── save-context/
│       ├── dev-engineering/       # справочный, вне ручного QA-контура
│       ├── product-management/    # справочный
│       ├── research-knowledge/    # справочный
│       └── token-efficient/       # справочный, стиль ответов
├── docs/                      # общие стандарты компании (методология QA)
├── checklists/                # общий чек-лист компании
├── templates/                 # общие шаблоны компании
├── local/                     # НЕ в git (кроме README.md) — адаптация продуктовой команды
│   ├── docs/                  #   документация конкретного продукта
│   ├── checklists/            #   override чек-листа
│   └── templates/             #   override шаблонов
├── results/                   # результаты работы агента
│   ├── requirements-review/
│   └── test-cases/
└── .context-cache/            # кэш сессий, TTL 2 дня (в .gitignore)
```
