# Чек-лист ревью требований (Shift Left)

**Назначение:** проверка качества требований до начала разработки/тестирования. Применяется к задаче в трекере, разделу ЧТЗ, ТЗ или произвольному тексту требований.

**Статусы пунктов:** ✅ ОК | ⚠️ Частично (нужно уточнение) | ❌ Отсутствует | ➖ Неприменимо

**Правило выхода:** по каждому ⚠️ и ❌ формируется вопрос аналитику/постановщику. Задача не берётся в тест-дизайн, пока не закрыты блокирующие пункты (помечены 🔴).

---

## 0. Подготовка: извлечение и нумерация требований

Выполняется всегда. Если ФТ/AC уже проставлены — только сверка, если нет — извлекаем сами.

* [ ] 0.1. 🔴 Определён формат постановки: явные ФТ/AC ↔ сплошной текст ТЗ ↔ смешанный
* [ ] 0.2. 🔴 Из текста выделены атомарные требования (одно требование = одна проверяемая мысль, без «и/или» внутри)
* [ ] 0.3. Каждому требованию присвоен идентификатор по схеме проекта: БТ-XXX → ФТ-XXX.N → КП/ТК-XXX.N.M
* [ ] 0.4. Неявные требования вытащены в явные (слова «должна», «обеспечивает», «автоматически», «недоступно», «только», «не более» — маркеры скрытых ФТ)
* [ ] 0.5. Для каждого ФТ сформулирован хотя бы один КП (критерий приёмки) в формате «Дано / Когда / Тогда» или «Действие → Ожидаемый результат»
* [ ] 0.6. Требования из таблиц, схем и приложений (структуры данных, перечни атрибутов) тоже пронумерованы, а не только текст
* [ ] 0.7. Зафиксировано, что в задаче НЕ входит в объём (out of scope)

## А. Бизнес-ценность

* [ ] А1. Проблема/цель сформулирована (зачем делаем)
* [ ] А2. Метрика успеха определена (как поймём, что сработало)
* [ ] А3. Целевой пользователь/роль указаны
* [ ] А4. Цена бездействия понятна (что будет, если не сделать)
* [ ] А5. Приоритет требования понятен (must / should / could)

## Б. Функциональные требования

* [ ] Б1. 🔴 Happy Path описан полностью (от входа до результата)
* [ ] Б2. Альтернативные сценарии описаны
* [ ] Б3. 🔴 Негативные сценарии описаны: что делает система при ошибке (блокирует? предупреждает? какой текст сообщения?)
* [ ] Б4. Граничные условия определены (мин/макс значений, длины полей, размеры файлов, количество элементов)
* [ ] Б5. 🔴 Бизнес-логика однозначна (нет «и т.д.», «по аналогии», «при необходимости» без расшифровки)
* [ ] Б6. Обязательность полей/атрибутов указана явно для каждого атрибута
* [ ] Б7. Для вычисляемых/автозаполняемых полей описаны: формула/источник, момент пересчёта, доступность ручного редактирования
* [ ] Б8. Для зависимых полей и справочников описано поведение при изменении родителя (пересчёт, очистка, блокировка)
* [ ] Б9. Поведение при редактировании и удалении описано, а не только при создании (в т.ч. каскадные эффекты и связанные сущности)
* [ ] Б10. Уникальность идентификаторов/номеров описана с учётом удалённых записей
* [ ] Б11. Состояния/статусы объекта и разрешённые переходы между ними определены

## В. Данные и связи (специфика системы)

* [ ] В1. 🔴 Полный перечень атрибутов есть: название, тип, обязательность, источник (ручной ввод / справочник / вычисляемое)
* [ ] В2. Связи с другими сущностями определены: кардинальность (1:1, 1:N, N:M), обязательность связи, что автозаполняется при создании связи
* [ ] В3. Поведение при разрыве/изменении связи описано (что происходит с автозаполненными атрибутами)
* [ ] В4. Справочники НСИ: указан конкретный справочник, его состав или источник; для зависимых — правило фильтрации
* [ ] В5. Форматы данных однозначны (даты, координаты, единицы измерения, разделители)
* [ ] В6. Геоданные: тип геометрии (точка/линия/полигон), система координат, правила ручного ввода координат
* [ ] В7. Миграция/начальные данные: понятно, откуда берутся существующие данные и как обрабатываются неполные записи

## Г. Интеграции и обмен данными

* [ ] Г1. Для каждой интеграции указаны: направление, формат, структура, частота/триггер обмена
* [ ] Г2. Поведение при недоступности внешней системы описано (справочники, геосервисы, платёжные шлюзы и т.п.)
* [ ] Г3. Обработка ошибок и невалидных ответов внешней системы описана
* [ ] Г4. Маппинг полей нашей системы ↔ внешней задан таблицей, без пропусков

## Д. Нефункциональные требования

* [ ] Д1. Производительность: время отклика, объёмы (учитывать реальные объёмы: сотни тысяч записей, км линий)
* [ ] Д2. Роли и доступы: какая роль что видит/редактирует именно в этой функции
* [ ] Д3. Совместимость: браузеры, мобильная версия — входит ли адаптив в объём задачи
* [ ] Д4. Логирование/аудит изменений: нужно ли фиксировать, кто и когда изменил
* [ ] Д5. Usability: сообщения об ошибках, подтверждения деструктивных действий, индикация загрузки

## Е. Тестируемость и реализуемость

* [ ] Е1. 🔴 Каждое ФТ проверяемо: можно придумать тест с однозначным ожидаемым результатом
* [ ] Е2. Тестовые данные доступны или понятно, как их создать (включая связанные сущности-предусловия)
* [ ] Е3. Тестовая среда доступна, интеграции на стенде замоканы или доступны
* [ ] Е4. Влияние на смежные модули оценено (что регрессировать)
* [ ] Е5. Стратегия проверки понятна (UI / API / БД / отчёт / выгрузка)
* [ ] Е6. Понятно, как проверить «невидимую» логику (автопересчёты, фоновые задания, расписания)

## Ж. Консистентность и трассируемость

* [ ] Ж1. Нет противоречий с другими требованиями ЧТЗ и уже реализованным поведением
* [ ] Ж2. Нет дублирования с существующими ФТ (проверить по реестру тест-кейсов)
* [ ] Ж3. Терминология едина и совпадает с глоссарием проекта (сокращения и термины домена)
* [ ] Ж4. 🔴 Трассируемость выстроена: каждое БТ покрыто ≥1 ФТ, каждое ФТ — ≥1 КП, каждый КП — ≥1 ТК
* [ ] Ж5. Обратная трассируемость: нет «висячих» ТК без требования (иначе требование потеряно или ТК лишний)
* [ ] Ж6. Ссылки на разделы ЧТЗ/макеты/схемы указаны и актуальны
