# Проектирование тест-кейсов

Соответствует этапу «Написание тест-кейсов» в Shift Left (`docs/Testirovanie-Shift-Left.md`).

## Входные данные
1. Требования: файл в `local/docs/` или `docs/`, либо готовый реестр из
   `results/requirements-review/` (результат навыка `testing-requirements`). Если реестр
   есть — используй его нумерацию БТ/ФТ как есть, не перенумеровывай.
2. Шаблон структуры кейса — сначала проверь `local/templates/test-case-template.md`
   (адаптация этой команды), иначе используй общий `templates/test-case-template.md`.
3. Правила написания — сначала проверь `local/docs/Napisanie-test-kejsov.md`, иначе общий
   `docs/Napisanie-test-kejsov.md`: обязательные атрибуты и правила формулировок (заголовок,
   шаги, ожидаемый результат и т.д.). Прочитай перед началом работы, это основной источник
   требований к качеству формулировок, не только шаблон.

Если использовал local-override шаблона или правил, скажи об этом тестировщику (пометка 🏠).

## Схема нумерации (БТ → ФТ → ТК)

Единая для всего проекта схема — та же, что в `checklists/requirements-checklist.md` (п.0.3)
и в примере `templates/test-case-example.md`: `БТ-XXX → ФТ-XXX.N → ТК-XXX.N.M`.
Позитивный/негативный сценарий — это значение поля **«Сценарий»** внутри кейса, а не
суффикс в ID (никаких `-P1`/`-N1`).

1. **Явно выдели требования.** Если пришёл готовый реестр БТ/ФТ из
   `results/requirements-review/` — используй его номера как есть.
   Если реестра нет — выдели требования сам и присвой номера по этой же схеме
   (БТ-XXX, у каждого — ФТ-XXX.1, ФТ-XXX.2, …). Если непонятно, с какого номера БТ
   начинать (чтобы не пересечься с уже занятыми в проекте), — спроси тестировщика,
   не угадывай. Покажи получившийся реестр в чате перед проектированием кейсов.
2. **Покрой каждое ФТ:**
   - минимум 1 кейс со «Сценарий: Позитивный» (основной сценарий);
   - минимум 1 кейс со «Сценарий: Негативный» (невалидные данные, граничные значения, ошибки);
   - если требование не тестируемо или для него не придумать негатив — не выдумывай,
     а зафиксируй это в разделе «Непокрытые требования» с причиной.
3. **Присвой ID кейсам:** `ТК-<ФТ>.<n>` по порядку внутри ФТ (например, `ТК-102.1.1`,
   `ТК-102.1.2`, `ТК-102.1.3` — все кейсы ФТ-102.1, вперемешку позитивные и негативные,
   различай их по полю «Сценарий», не по номеру). Это рабочий ID до выгрузки в Test IT;
   реальный `ID (Test IT)` присвоит сам Test IT при создании (навык `testit-sync`).

## Алгоритм
1. Построй реестр БТ/ФТ и покажи тестировщику (см. «Схема нумерации» выше).
2. Покрой каждое ФТ позитивными и негативными кейсами (см. выше).
3. **Оформи кейсы по шаблону** (`templates/test-case-template.md`) и правилам
   (`docs/Napisanie-test-kejsov.md`).
4. **Приоритет** — по риску сценария (Самый низкий…Самый высокий); тест-кейсы по
   рисковым/критичным сценариям — всегда «Самый высокий».
5. **Статус** нового кейса — «Не готов», пока не проверен тестировщиком; «Статус
   автоматизации» — «Не автоматизирован» по умолчанию.
6. **Самопроверка (обязательна, не пропускай)** — см. раздел ниже. Прогони черновик через
   все проверки до сохранения; найденное исправляй сразу, не сдавай «сырой» черновик.
7. **Сохрани** в `results/test-cases/YYYY-MM-DD-<тема>.md`: сначала таблица трассировки,
   затем кейсы, сгруппированные по функциональным блокам (не вперемешку), в конце — блок
   «Самопроверка агента».
8. В чат выведи только таблицу трассировки, статистику и итог самопроверки, не сами кейсы.

## Таблица трассировки (обязательна в начале файла)

Формат — компактный обзор покрытия по БТ/ФТ, по образцу `templates/test-case-example.md`
(тот файл показывает именно этот стиль таблицы, не формат отдельного кейса — за полным
кейсом смотри `test-case-template.md`). Шаги/ожидаемый результат в таблицу не дублируй —
они только в теле кейса ниже, чтобы не раздувать файл.

```markdown
### БТ-102: <краткое название бизнес-требования>

| ТК | Название | Сценарий |
|---|---|---|
| ТК-102.1.1 | Отображение по умолчанию | Позитивный |
| ТК-102.1.2 | Ошибка при невалидных данных | Негативный |
```

Требования без кейсов (см. «Непокрытые требования») в эту таблицу не включай — им отдельный
раздел с причиной.

## Самопроверка (обязательный финальный шаг)

Тестировщики регулярно забывают перепроверить собственный набор кейсов на полноту и
единообразие полей — агент должен делать это каждый раз механически:

1. **Полнота покрытия.** Для каждого ФТ из реестра: есть хотя бы 1 кейс со
   «Сценарий: Позитивный» И хотя бы 1 со «Сценарий: Негативный» в теле файла, ИЛИ
   требование явно перечислено в «Непокрытые требования» с причиной. Ни одно ФТ не
   должно молча отсутствовать и там, и там.
2. **Обязательные поля.** У каждого кейса на месте все поля из шаблона: ID (Test IT),
   Требование, Тег, **Сценарий**, Приоритет, Статус, Статус автоматизации, Заголовок,
   Предусловия, Шаги, Ожидаемый результат на каждый шаг. Пустых/пропущенных полей нет
   (особенно легко забыть «Сценарий» — сверь явно).
3. **Таблица трассировки совпадает с телом.** Каждый ТК-ID, упомянутый в таблице, есть в
   теле файла с тем же «Сценарий», и наоборот.
4. **Эффект пестицида.** Перечитай шаги всех кейсов на предмет зашитых статических данных
   (конкретный логин/email/пароль/имя файла) — по правилам `docs/Napisanie-test-kejsov.md`
   их там быть не должно. Найденное — перепиши обобщённо, а не оставляй как есть.
5. **Формулировки шагов.** Один шаг — одно действие, безличный глагол, есть ожидаемый
   результат у каждого шага.

В конце файла добавь блок:

```markdown
---

## Самопроверка агента
- ФТ в реестре: N, покрыто (позитив+негатив): N/N, в «Непокрытые» с причиной: N
- Кейсов создано: N, все обязательные поля (включая «Сценарий») на месте: да/нет
- Таблица трассировки совпадает с телом файла: да/нет
- Проверка на зашитые статические данные (эффект пестицида): чисто / найдено и исправлено N
```

Если хотя бы один пункт — «нет» или расхождение не устранено, не показывай набор кейсов
как готовый: исправь и перепроверь снова.

## Правила формулировок (из docs/Napisanie-test-kejsov.md — не нарушай)
- Заголовок: чёткий, краткий, без шагов и ожидаемого результата внутри.
- Один шаг — одно действие (один метод автотеста), безличный глагол:
  «нажать», «ввести», «перейти» — не «нажмите», «введите».
- Не детализируй шаги избыточно: «ввести в поле число 12», а не «нажать клавишу ‘1’,
  затем клавишу ‘2’».
- **Не зашивай в шаги статические данные** (конкретные логины, пароли, имена файлов) —
  это эффект пестицида. Пиши обобщённо и с абстрактными/условными значениями:
  «ввести число 12», «ввести значение из предусловия» — а не привязывайся к
  конкретному реальному логину/паролю.
- Предусловия не должны содержать данные для авторизации — только ссылку на инструкцию,
  где они описаны.
- Ожидаемый результат — у каждого шага, кратко, без избыточного описания.
- Кейс должен быть независим от других кейсов (без ссылок «см. ТК-…» как предусловия).
- Без скриншотов в кейсах на функциональность (только в кейсах на отображение форм/страниц).

## Другие правила
- Шаги кейса — только из того, что известно о системе из `local/docs/` / `docs/`. Если
  непонятно, как устроен экран/поле/процесс, — спроси тестировщика, не сочиняй интерфейс.
- Негативные кейсы выбирай по классам: пустые/невалидные значения, границы, дубли,
  права доступа, прерывание процесса — что применимо к требованию.
- Если готовишь кейсы для конкретной User Story/задачи — сгруппируй их так, чтобы
  название группы совпадало с названием US, а внутри — с названием задачи (это нужно
  для `testit-sync`: секция в Test IT = US, сьют = задача, см. `docs/Testirovanie-Shift-Left.md`).
- После сохранения предложи (не выполняй сам) выгрузку в Test IT навыком `testit-sync`.
