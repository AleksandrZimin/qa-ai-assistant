# ИИ-ассистент по тестированию веб-приложений

Рабочее пространство для тестировщиков: клонируете репозиторий, открываете
Claude Code — и сразу пишете промпты. Агент уже настроен: знает методологию QA
компании (`docs/`), работает по чек-листам и шаблонам команды, умеет выгружать
кейсы в Test IT.

Компания работает в матричной структуре — много продуктовых команд поверх единых
стандартов. Этот шаблон поддерживает это из коробки: `docs/` — общие стандарты
компании (в git, обновляются централизованно), `local/` — адаптация конкретно
вашей команды/продукта (не в git, никогда не конфликтует с другими командами).
Подробнее — в [local/README.md](local/README.md).

**Язык общения с агентом:** по умолчанию русский. Если написать агенту на английском
или явно попросить («switch to English») — он продолжит на английском (включая
сохранённые файлы), пока не попросите вернуться обратно. Методология в `docs/` /
`checklists/` / `templates/` при этом остаётся на языке оригинала — агент переводит
её на лету в ответах, а не хранит два набора файлов.

## Быстрый старт

1. **Клонируйте репозиторий** и откройте папку в терминале или VS Code.
2. **Установите Claude Code** (если ещё нет): [инструкция](https://docs.anthropic.com/claude-code).
3. **Настройте Test IT** (нужно только для выгрузки кейсов):
   ```
   copy .env.example .env
   ```
   и заполните `TESTIT_URL`, `TESTIT_TOKEN`, `TESTIT_PROJECT_ID` в `.env`.
4. **Положите документацию своего продукта** в папку `local/docs/` (формат `.md`) —
   она приоритетнее общих стандартов и не уйдёт в git. Если у вашей команды свой
   чек-лист или шаблон кейса — положите их в `local/checklists/` / `local/templates/`
   с теми же именами, что в общих папках.
5. **Запустите** `claude` в папке проекта и пишите промпт.

## Что умеет агент

| Задача | Пример промпта |
|---|---|
| Тестирование требований по чек-листу (статусы по пунктам, вопросы аналитику, итоговый вердикт) | «Протестируй требования из local/docs/auth-requirements.md» |
| Тест-кейсы (позитивные + негативные) на каждое требование | «Покрой требования раздела 2 тест-кейсами» |
| Создание кейсов в Test IT (секция = US, сьют = задача) | «Выгрузи кейсы из последнего файла в Test IT» |
| Ответы по документации продукта и методологии QA | «Сколько действует ссылка восстановления пароля?», «Что делать при инциденте в проде?» |
| Сохранение контекста между сессиями | «Сохрани контекст» (и автоматически после крупных задач) |

Результаты агент складывает в `results/`: отчёты по требованиям — в
`requirements-review/`, тест-кейсы — в `test-cases/`.

Отчёт по требованиям по регламенту компании публикуется комментарием в задаче
YouTrack — агент такой интеграции не имеет, поэтому готовит текст в файле так,
чтобы его можно было скопировать в комментарий как есть.

За рамками ручного тестирования (код, автоматизация, PRD, глубокие исследования)
у агента есть справочные скиллы — они не включаются сами по себе, только если вы
явно попросите о таком выходящем за рамки ручного тестирования вопросе.

## Четыре правила работы с агентом

1. **Агент не выдумывает.** Если ответа нет ни в `local/`, ни в `docs/`, он скажет об
   этом и задаст вопрос. Помечает источник: 🏠 — из вашего локального контекста,
   📁 — из общих стандартов компании, 🧠 — из общих знаний.
2. **`local/` всегда приоритетнее `docs/`.** Если вы положили в `local/docs/`,
   `local/checklists/` или `local/templates/` файл с тем же именем, что в общей папке —
   агент возьмёт ваш вариант. Ничего донастраивать не нужно.
3. **Контекст живёт 2 дня.** Итоги сессий сохраняются в `.context-cache/` и доступны
   агенту в следующих сессиях; через 2 дня очищаются автоматически. Вернулись к задаче
   на следующий день — просто продолжайте, агент вспомнит.
4. **Чем конкретнее промпт, тем лучше результат.** Указывайте файл с требованиями и
   раздел: «покрой кейсами раздел 1 из local/docs/auth-requirements.md», а не «напиши тесты».

## Настройка под свою команду

Общие стандарты компании (`docs/`, `checklists/`, `templates/`) менять не нужно — они
централизованные и обновляются для всех сразу. Все свои адаптации — продуктовую
документацию, другой чек-лист, другой шаблон кейса — кладите в `local/` (см.
[local/README.md](local/README.md)); эта папка не в git, поэтому ваши изменения не
конфликтуют с другими командами и не теряются при обновлении шаблона.

Как устроен агент внутри и как расширять его умения — в [HOW-IT-WORKS.md](HOW-IT-WORKS.md).
